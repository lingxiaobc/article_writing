# 文章写作项目 — 项目记忆

## 项目定位
本项目专用于公众号文章的撰写、优化与配图管理。所有技能、命令和智能体均存放于 `.claude/` 目录。

---

## 核心技能

| 技能 | 触发方式 | 说明 |
|------|----------|------|
| `optimize-article` | 提供文章链接/路径，或说"优化文章"、"润色" | 全面优化公众号文章，含结构、语言、逻辑，并为每个大板块自动生成手绘信息图 |
| `physics-lecture` | 提供物理草稿/大纲，或说"写物理讲义" | 针对40-50分学生，以费曼风格编写高中物理讲义 |
| `cover-generator` | 说"请为{文章路径}生成封面"，或说"生成封面图" | 根据文章生成公众号封面图：含IP形象（Q版小猫）+ 标题艺术字，简约彩色风格，2.35:1比例 |

---

- **技能逻辑**：`.claude/skills/physics-lecture/SKILL.md`

## 重要约定

- **API Key**：Gemini API Key 存放于 `.claude/settings.local.json` 的 `env.GEMINI_API_KEY`，脚本自动读取，无需手动配置
- **保存策略**：优化/生成完成后，自动在 `draft/` 目录下创建以文章标题命名的文件夹，文章和配图分别存放在该目录及其 `image/` 子目录中
- **语言跟随**：中文文章输出中文，英文文章输出英文
- **目录命名规范**：去除 Windows 不允许的字符（`\ / : * ? " < > |`），长度不超过 50 个字符

## 文本强调规范（适用于所有技能与任务）

### 核心原则
所有输出的文章、讲义、文档，必须遵循以下文本强调规范，以提升可读性和重点突出效果。

### 加粗规范
对以下内容使用 `**加粗**` 格式：

1. **核心观点与结论**
   - 引出结论的关键句、核心论点、重要判断
   - 示例：`经过分析，我发现**问题的根源在于沟通不畅**。`、`**这才是真正的价值所在**。`

2. **重点词语与关键短语**
   - 需要读者重点关注的词或短语、关键概念
   - 示例：`这里的**核心问题**在于...`、`**最重要的一点**是...`

3. **强调性词汇**
   - 提示性词汇：**注意**、**重点**、**关键**、**核心**、**必须**、**千万**
   - 情绪性词汇：**太棒了**、**震惊**、**意外**、**惊喜**
   - 示例：`**注意**，这里有个常见误区。`

4. **行动召唤**
   - 结尾的行动呼吁句或关键指令，整句或核心动词加粗
   - 示例：`**现在就开始行动吧！**`、`你需要**立即检查**这个配置。`

### 高亮规范
对以下内容使用 `==高亮==` 格式（Markdown 高亮语法）：

1. **精华语句**
   - 全文最核心的观点、金句、可以独立传播的句子
   - 必须单独成段，前后各空一行
   - 示例：
     ```

     ==真正的价值不在于你做了多少，而在于你解决了什么问题。==

     ```

2. **重点总结**
   - 章节核心结论、方法论总结、关键要点
   - 必须单独成段，前后各空一行
   - 示例：
     ```

     ==记住：先理解问题，再寻找方案，最后才是执行。==

     ```

3. **核心公式或定义**（适用于物理讲义）
   - 关键物理定律、核心公式的文字表述
   - 必须单独成段，前后各空一行
   - 示例：
     ```

     ==加速度的本质：描述速度变化的快慢。==

     ```

### 使用约束
- **加粗克制使用**：每个段落加粗不超过 1-2 处，避免全文到处加粗导致视觉噪音
- **高亮极度克制**：全文高亮不超过 3-5 处，只用于最精华的内容
- **不重复强调**：同一个词或短语，全文只在第一次出现时强调，后续不再重复
- **不叠加使用**：不要同时使用加粗和高亮（`**==文本==**`），二选一即可

### 示例对比

**优化前：**
```
这个方法很重要。但是很多人不知道。你需要注意这一点。真正掌握需要反复练习。
```

**优化后：**
```
这个方法很重要。但是很多人不知道。你需要**注意**这一点。

==真正掌握这个方法，需要反复练习。==
```

## 长文档写入规范（适用于所有技能与任务）

### 核心原则

**禁止边思考边写入**：当需要写入的文档或文字内容**超过 200 行**时，严格遵循以下流程，不得边思考边写入文件。

### 标准流程

1. **第一步：编写框架**
   - 先构思完整的文档框架（大纲、章节结构、核心要点）
   - 将框架写入文件，作为文档骨架
   - 等待用户审核框架，确认无误后再继续

2. **第二步：逐板块填充内容**
   - 按框架顺序，逐个板块思考并填充内容
   - 每完成一个板块，立即写入文件
   - 不要一次性思考所有板块，避免"手没跟上脑子"

3. **第三步：全文检查**
   - 所有板块填充完成后，**必须**通读全文
   - 检查板块之间的衔接是否自然
   - 检查逻辑是否合理、前后是否一致
   - 发现问题立即修正

### 子代理并行处理

**适用场景**：当任务中有多个独立的板块（板块之间关联性不强、可独立完成）时，可使用子代理并行处理。

**操作方法**：
- 将独立板块分配给不同的子代理
- 每个子代理独立完成自己的板块
- 所有子代理完成后，主进程汇总结果
- 汇总后**必须**进行全文检查，确保板块之间衔接合理

**示例场景**：
- 优化多篇独立文章
- 生成多个独立的配置文件
- 编写多个独立的功能模块文档

### 判断标准

**如何判断是否超过 200 行？**
- 预估文档最终行数（包括空行、代码块、列表等）
- 如果不确定，按"可能超过 200 行"处理
- 宁可多一步框架审核，也不要冒险边想边写

**如何判断板块是否独立？**
- 板块之间没有强依赖关系（A 板块不需要 B 板块的结果）
- 板块可以独立理解和完成
- 板块的顺序可以调整而不影响整体逻辑

### 违规后果

如果违反此规范导致文档出现以下问题：
- 内容不完整或突然中断
- 板块之间逻辑不连贯
- 前后矛盾或重复

**必须**：
1. 向用户说明问题原因
2. 按标准流程重新编写
3. 将此次错误记录到项目记忆中，避免再犯

## 目录用途说明

| 目录 | 说明 |
|------|------|
| `draft/` | 文章草稿存储目录，每篇文章一个独立子文件夹（以文章标题命名），包含优化后文章及配套图片 |
| `draft/[文章标题]/` | 单篇文章的独立目录，包含文章文件和 `image/` 子目录 |
| `draft/[文章标题]/image/` | 该文章的所有配图存放目录 |
| `generated_images/` | 图片生成默认输出目录（已废弃，保留兼容） |
| `.obsidian/` | Obsidian 笔记软件配置，用于 Markdown 编辑与预览 |
