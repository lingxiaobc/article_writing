# 高中物理讲义编写技能

## 核心目标

> 让高中物理成绩在 **40-50分** 区间的学生：
> 1. **看得懂** — 消除认知障碍，理解知识本质
> 2. **学得会** — 掌握解题思路与核心方法
> 3. **想看下去** — 语言有趣，节奏有吸引力

---

## 输入处理

根据 `$ARGUMENTS` 或用户上下文判断输入类型：

- **本地草稿文件**（`.md`、`.txt`、`.docx` 路径）：使用 `Read` 工具读取草稿内容。
- **网页链接**：使用 `WebFetch` 工具抓取内容。
- **直接粘贴内容**：以用户粘贴的草稿/知识点作为来源。
- **仅描述知识点**（如"牛顿第二定律"）：以该知识点为主题从零创作。

### 自动识别草稿触发

即使用户**没有明确说"写讲义"**，只要满足以下任一条件，也应自动触发本技能：

1. 用户粘贴了含有物理教学内容的文字（包含物理术语、公式推导、例题、受力分析、知识点说明等特征词汇）
2. 用户提供了文件路径，读取后判断为物理教学草稿或教案
3. 用户在消息中提及"物理草稿"、"讲义草稿"、"教案"、"知识点整理"等关键词并附带内容

**识别后的处理方式**：先向用户确认意图，说：
> "我看到你发送了一份物理草稿，是否需要我将它编写成适合40-50分学生阅读的讲义？"

用户确认后，立即进入编写工作流程。

若未提供任何输入，询问用户：
"请提供物理草稿文件路径、知识点描述，或直接粘贴草稿内容，我来帮你编写成讲义。"

## 编写工作流程

### 第零步 — 说明流程并确认

在执行任何操作之前，向用户说明本次任务的完整流程与目标，格式如下：

> **即将执行：高中物理讲义编写**
>
> **输入来源：** [本地草稿文件：路径 / 网页链接 / 直接粘贴内容 / 知识点描述：xxx]
>
> **执行步骤：**
> 1. 读取草稿内容，提取核心知识点、例题与风格特征
> 2. 结合学生画像（40-50分学段），分析学习痛点、认知负荷、前置知识与常见错误
> 3. 按费曼风格写作原则进行讲义创作
> 4. 输出完整讲义（含章节标题、学习目标、概念讲解、例题、易错提醒等模块）
> 5. 质量自检，确认无跳步、无拔高、无捏造
> 6. 询问是否保存文件
>
> **目标：** 让40-50分学生看得懂、学得会、想看下去。
>
> 请确认是否按此流程执行，或告知需要调整的地方。

**等待用户确认后，方可进入第一步。**

---

### 第一步 — 获取草稿内容
根据输入类型读取草稿，提取：
- 核心知识点列表
- 已有例题与解析
- 草稿的语言风格特征（用于后续保持风格一致性）

### 第二步 — 学生认知诊断
读取并应用 [student-profile.md](student-profile.md) 中的学生画像，分析：

1. **本章节的学习痛点** — 该知识点哪里最容易让40-50分学生卡住？
2. **认知负荷评估** — 草稿中哪些概念对目标学生而言认知负荷过重？
3. **前置知识检查** — 学生是否具备理解本章的先备知识？
4. **常见错误模式** — 该分段学生最容易在哪里犯什么样的错误？

输出简要的**认知诊断表**（内部分析，不出现在讲义正文中）。

### 第三步 — 应用讲义写作原则
严格遵循 [principles.md](principles.md) 中的写作原则进行创作：

- 费曼风格：用日常语言重新解释物理概念
- 层次结构：概念 → 直觉理解 → 公式推导 → 例题 → 变式练习
- 语言风格：与草稿保持一致，不做风格大幅偏移
- 节奏控制：设置悬念、制造好奇、给学生"啊，原来是这样"的感觉

若需统计草稿字数与结构，可运行：
```bash
python .claude/skills/physics-lecture/scripts/analyze_draft.py <文件路径>
```

### 第四步 — 编写讲义正文
按照 [assets/lecture_template.md](assets/lecture_template.md) 的结构模板输出完整讲义，必须包含：

| 模块 | 说明 |
|------|------|
| 章节标题 | 有吸引力，点明学习价值 |
| 学习目标 | 用学生语言写，具体可检验 |
| 痛点引入 | 先戳痛点，再给解法 |
| 概念讲解 | 费曼风格，日常类比优先 |
| 公式解读 | 逐项拆解，说人话 |
| 例题精讲 | 分步骤，说出每步"为什么" |
| 易错提醒 | 针对40-50分学生的典型错误 |
| 小结回顾 | 用一句话或思维导图收束 |

### 第五步 — 质量自检
编写完成后，对照以下标准自我检查：

- [ ] 每个物理概念是否都有日常类比或具体情境？
- [ ] 公式是否都有逐项的文字解释？
- [ ] 是否存在"跳步"（对目标学生而言理解不到的逻辑跳跃）？
- [ ] 语言风格是否与草稿保持一致？
- [ ] 例题讲解步骤是否完整，每步都有说明？
- [ ] 是否有至少一处"费曼式追问"（你真的理解了吗？让我们来验证一下）？

### 第六步 — 输出与保存
输出完整讲义后，若来源为本地文件，询问用户：
"是否将讲义保存为新文件（如 `[章节名]-讲义.md`），还是覆盖原草稿文件？"

## 执行约束

- **不拔高**：不为40-50分学生引入超纲内容或竞赛技巧。
- **不简化过度**：保持物理概念的准确性，不因"简单化"而引入错误认知。
- **保留草稿风格**：语言风格与草稿保持一致，不做无必要的文风转变。
- **不捏造例题答案**：所有例题解答须经过物理验算，数值正确。
- **标注不确定处**：若草稿中有模糊表述，标注说明，而非擅自猜测改写。
- **层次清晰**：每个知识点严格遵循"是什么 → 为什么 → 怎么用"三层逻辑。
- **终端命令须事先说明**：每次执行 Bash 命令前，须向用户说明该命令的具体用途，不得静默执行。
- **不确定即暂停汇报**：执行过程中若遇到任何不确定、没把握的情况，必须立即暂停并向用户汇报，等待指示后再继续。

## 长文档写入规范

**适用场景**：当编写的讲义预计**超过 200 行**时，必须遵循以下流程。

### 标准流程

1. **第一步：编写框架**
   - 先构思完整的讲义框架（知识点结构、例题分布、练习题安排）
   - 将框架写入文件，包含：
     - 讲义标题
     - 各知识点的二级标题（`##`）
     - 各知识点的核心要点（简要列出）
     - 例题和练习题的占位符
   - 告知用户："框架已完成，请审核。确认无误后将逐板块填充内容。"
   - 等待用户确认

2. **第二步：逐板块填充内容**
   - 按框架顺序，逐个知识点思考并填充内容
   - 每完成一个知识点，立即使用 `Edit` 工具写入文件
   - 不要一次性思考所有知识点，避免"手没跟上脑子"
   - 每完成 2-3 个知识点，向用户汇报进度

3. **第三步：全文检查**
   - 所有知识点填充完成后，**必须**通读全文
   - 检查知识点之间的衔接是否自然
   - 检查逻辑是否合理、前后是否一致
   - 检查例题答案是否正确
   - 检查是否有遗漏的练习题
   - 发现问题立即修正

### 子代理并行处理

**适用场景**：当讲义中有多个独立知识点（知识点之间关联性不强）时，可使用子代理并行处理。

**操作方法**：
- 将独立知识点分配给不同的子代理
- 每个子代理独立完成自己的知识点内容
- 所有子代理完成后，主进程汇总结果
- 汇总后**必须**进行全文检查，确保知识点之间衔接合理

**注意事项**：
- 如果知识点之间有强逻辑关系（如递进、因果、前置依赖），不适合并行处理
- 例题和练习题的难度应保持一致，避免子代理之间难度差异过大
